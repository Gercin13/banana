# Nano Studio

Генератор изображений на Google Gemini (**Nano Banana**) с простым интерфейсом и **прочной** архитектурой: постоянный Node-сервер + хранение на диске + история. Изображения отдаются по ссылкам (не base64), поэтому 2K/4K и пачки работают надёжно. Контейнеризован — запускается одинаково локально, на VPS и на любом хостинге контейнеров.

## Возможности
- Промпт с опциональным **голосовым вводом** (Chrome/Edge).
- Все популярные **соотношения сторон**: 1:1 … 21:9.
- **1–4 изображения** за раз (параллельно).
- **Разрешение 1K / 2K / 4K**.
- **Референсы для консистентного персонажа:** «Лицо / личность» + «Остальное» (позы/одежда/стиль), суммарно до 14.
- Два режима: **Черновик** (Nano Banana 2) / **Качество** (Nano Banana Pro).
- **Хранение результатов + История** (переживает перезагрузку страницы и рестарт сервера).

## Запуск — вариант 1: Docker (рекомендуется, самый надёжный)
Нужен установленный Docker Desktop / Docker Engine.
```bash
cp .env.example .env          # впишите GEMINI_API_KEY
docker compose up --build     # → http://localhost:3000
```
Изображения и история сохраняются в папку `./data` на вашем компьютере (том примонтирован), поэтому не теряются между перезапусками. Остановить: `Ctrl+C` (или `docker compose down`).

## Запуск — вариант 2: локально через Node (без Docker)
Нужен Node.js ≥ 18.
```bash
npm install
cp .env.example .env          # впишите GEMINI_API_KEY
npm start                      # → http://localhost:3000
```
Данные — в папке `./data`.

## Ключ API
Google AI Studio → **Get API key** — https://aistudio.google.com. Вставьте в `.env`:
```
GEMINI_API_KEY=ваш_ключ
```
Ключ используется **только на сервере** и никогда не попадает в браузер.

## Деплой на хостинг (когда захотите открыть доступ извне)
Это обычный Node-сервер, поэтому подойдёт любой хостинг, где работает контейнер или Node-процесс. Везде: (1) задайте переменную окружения `GEMINI_API_KEY`; (2) для сохранности истории подключите постоянный диск, смонтированный в `/app/data`.

- **Railway / Render / Fly.io:** создайте проект из репозитория (или `fly launch`) — сборка пойдёт по `Dockerfile`. Добавьте env `GEMINI_API_KEY`. Прикрепите volume на `/app/data` (Railway Volume / Render Disk / Fly Volume) — иначе история очистится при передеплое.
- **Свой VPS:** установите Docker → `docker compose up -d --build`. Данные в `./data` на сервере. Для домена и HTTPS поставьте сверху reverse-proxy (Caddy или Nginx).

## Архитектура
```
nano-studio/
├── public/               статичный фронтенд (vanilla JS, без сборки)
│   ├── index.html  styles.css  app.js
├── lib/
│   ├── engine.js         GenerationEngine (Nano Banana) — сменный движок
│   ├── store.js          хранение: изображения на диск + записи истории
│   └── handler.js        валидация → генерация → сохранение → URL
├── server.js             Express: статика, /images, API (/api/*)
├── Dockerfile  docker-compose.yml  .dockerignore
├── .env.example  package.json
└── data/                 (создаётся в рантайме) images/ + records/   [gitignored]
```
Почему это «не хрупко»:
- **Ключ только на сервере.** Браузер получает лишь ссылки на изображения.
- **URL вместо base64** — нет лимитов на размер ответа; 4K и пачки работают.
- **Движок за интерфейсом** (`lib/engine.js`) — заменяется на FLUX / локальный ComfyUI без правок UI.
- **Хранение за интерфейсом** (`lib/store.js`) — сейчас диск; заменяется на S3/R2 + БД без правок роутов.

## Рост до мультипользователя (заложенные швы)
| Слой | Сейчас | Потом | Где менять |
|---|---|---|---|
| Аутентификация | один юзер (`local`) | сессии / OAuth / JWT | `getUserId()` в `server.js` |
| Хранение | диск (в записях есть `owner`) | S3/R2 + Postgres, скоуп по `owner` | `lib/store.js` |
| Квоты / лимиты | нет | на пользователя | `lib/handler.js` |

## API
- `GET /api/health` — возможности (модели, размеры, лимиты).
- `POST /api/generate` — тело `{prompt, aspectRatio, size, count, tier, faceRefs[], otherRefs[]}` → `{id, images:[{id,url}], model}`.
- `GET /api/history?limit=200` — последние генерации.
- `DELETE /api/history/:id` — удалить генерацию и её файлы.
